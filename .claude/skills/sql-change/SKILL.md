# sql-change スキル

このプロジェクトはEF Core Migrationを使わず、SQLiteスキーマを直接SQLで管理している。
変更時は以下の手順を順番に実施すること。

## $ARGUMENTS

変更内容の説明（例: `T_Productにremarkカラムを追加`）

---

## 手順

### 1. 現状確認

- `sql.md` を読んで対象テーブル・Viewの現状スキーマを把握する
- `Data/` 配下の対応リポジトリクラスを確認する
- `Models/` 配下の対応モデルクラスを確認する

### 2. 影響範囲の特定

以下を確認し、変更が必要なファイルをリストアップする:

- 変更対象テーブルを参照する **View**（`sql.md` のView定義）
- 変更対象テーブルを使う **Dapper クエリ**（`Data/` 配下の `SELECT *` や列名直書き）
- 変更対象テーブルのモデルクラス（`Models/` 配下）
- CSV エクスポートや Excel 出力でそのテーブルを扱う箇所

### 3. DDL SQL の作成

SQLite の制約に注意:
- `ALTER TABLE ... ADD COLUMN` は可能（`DROP COLUMN` / カラム名変更は SQLite 3.35+ 以降）
- Viewの変更は `DROP VIEW IF EXISTS` → `CREATE VIEW` の順
- 外部キー制約変更はテーブル再作成が必要

### 4. sql.md の更新

スキーマ変更後、`sql.md` を最新の状態に更新する。
`sql.md` はスキーマの唯一の信頼できるドキュメントなので、必ず同期すること。

### 5. モデルクラスの更新（`Models/`）

追加カラムに対応するプロパティを追加する。
nullable 設定（`?`）は既存パターンに合わせる。

### 6. リポジトリの更新（`Data/`）

- `SELECT` クエリに新カラムを追加
- `INSERT` / `UPDATE` クエリに新カラムを追加
- Dapper の `Query<T>` のマッピングが正しいか確認

### 7. View 関連の更新

変更対象テーブルを参照するViewがある場合:
- `sql.md` のView定義を更新
- View を使うリポジトリ・クエリも合わせて更新

### 8. 確認チェックリスト

- [ ] `sql.md` が最新のスキーマを反映している
- [ ] 影響を受けるすべてのモデルクラスを更新した
- [ ] 影響を受けるすべてのリポジトリクエリを更新した
- [ ] View定義を更新した（該当する場合）
- [ ] 既存の動作に影響がないか確認した
